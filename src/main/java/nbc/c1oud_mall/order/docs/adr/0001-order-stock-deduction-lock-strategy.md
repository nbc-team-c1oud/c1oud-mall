# ADR-0001: 주문 생성 시 재고 차감 동시성 제어 전략 선택

* 상태: Accepted
* 일자: 2026-06-05
* 범위: 주문 BC / 상품 BC

## Context

주문 생성 API에서는 장바구니 상품을 기반으로 주문을 생성하면서 상품 재고를 선차감한다.

현재 주문 생성 흐름은 다음과 같다.

```text
주문 생성 요청
  → 장바구니 상품 조회
  → 상품 재고 차감
  → OrderItem 스냅샷 생성
  → Order 저장
  → Payment 사전등록
```

이때 동일한 상품에 대해 여러 사용자가 동시에 주문을 생성하면 재고 차감 경쟁이 발생할 수 있다.

방어해야 할 시나리오:

1. 동일 상품 동시 주문

    * 재고가 1개 남은 상품에 대해 여러 사용자가 동시에 주문 생성 요청을 보냄
    * 두 요청이 모두 재고가 충분하다고 판단하면 재고가 음수가 되거나 초과 주문이 발생할 수 있음

2. 복수 상품 주문 간 교차 락

    * 주문 A: 상품 1 → 상품 2 순서로 재고 차감
    * 주문 B: 상품 2 → 상품 1 순서로 재고 차감
    * 서로 다른 순서로 락을 획득하면 데드락 가능성이 있음

3. 결제 실패 또는 결제대기 주문 취소

    * 주문 생성 시 재고를 선차감하므로, 이후 결제 실패나 주문 취소 시 선차감 재고를 복구해야 함

## 선택지

### A. DB 비관적 락 적용 ✅ 채택

* 상품 조회 시 `PESSIMISTIC_WRITE`를 사용하여 `SELECT ... FOR UPDATE` 방식으로 상품 행 잠금
* 재고 차감 전 상품 행을 잠가 동일 상품에 대한 동시 차감을 순차 처리
* 복수 상품 주문 시 상품 ID 기준으로 정렬 후 차감하여 데드락 가능성 완화

예시:

```java
@Lock(LockModeType.PESSIMISTIC_WRITE)
@Query("SELECT p FROM Product p WHERE p.id = :productId")
Optional<Product> findByIdForUpdate(@Param("productId") Long productId);
```

```java
cartItems.sort(Comparator.comparing(cartItem -> cartItem.getProduct().getId()));
```

장점:

* 재고 차감 시점의 동시성 문제를 DB 레벨에서 직접 제어 가능
* 구현이 직관적이며 재고 부족 검증과 차감을 하나의 트랜잭션 안에서 처리 가능
* 현재 프로젝트 규모에서 Redis 등 추가 인프라 없이 적용 가능

단점:

* 동일 상품에 주문이 몰리면 락 대기 시간이 증가할 수 있음
* 트랜잭션 범위가 길어지면 락 보유 시간이 길어질 수 있음
* 복수 상품 주문에서는 락 획득 순서를 정하지 않으면 데드락 가능성이 있음

---

### B. 낙관적 락

* Product 엔티티에 `@Version` 필드를 추가하여 동시 수정 충돌 감지
* 재고 차감 시점에 버전 충돌이 발생하면 예외 처리 후 재시도 또는 실패 응답

장점:

* 평소에는 DB 락을 직접 잡지 않아 락 대기 부담이 적음
* 충돌 빈도가 낮은 경우 성능상 유리할 수 있음

단점:

* 재고 차감처럼 충돌 가능성이 높은 구간에서는 재시도 로직이 필요함
* 주문 생성, 재고 차감, 결제 사전등록 흐름에서 충돌 처리 정책이 복잡해질 수 있음
* 사용자 입장에서 주문 생성 실패 또는 재시도 응답 처리가 필요함

---

### C. Redis 분산 락

* 상품 ID를 기준으로 Redis Lock 획득 후 재고 차감
* 다중 인스턴스 환경에서 애플리케이션 레벨 동시성 제어 가능

장점:

* 다중 서버 환경에서도 상품 단위 잠금 가능
* DB 락보다 유연한 락 타임아웃 정책을 설계할 수 있음

단점:

* Redis 인프라 의존성이 추가됨
* 락 획득 실패, 타임아웃, 락 해제 실패에 대한 별도 정책이 필요함
* 현재 프로젝트 규모에서는 과도한 복잡도라고 판단

---

### D. 단순 재고 검증 후 차감

* 상품 조회 후 재고 수량을 확인하고 차감
* 별도 락 없이 JPA 변경 감지로 재고 변경 반영

장점:

* 구현이 가장 단순함

단점:

* 동시 주문 상황에서 초과 차감 가능성이 있음
* 재고 정합성을 보장하기 어려움
* 커머스 주문 생성 흐름에는 부적합하다고 판단

## Decision

**A안, DB 비관적 락을 채택한다.**

주문 생성은 재고 수량이 직접 변경되는 핵심 흐름이며, 초과 주문이 발생하면 주문·결제·재고 정합성이 모두 깨질 수 있다.

따라서 상품 재고 차감 전 `Product` 행에 `PESSIMISTIC_WRITE` 락을 적용하여 동일 상품에 대한 재고 차감을 순차 처리한다.

또한 한 주문에서 여러 상품을 차감할 수 있으므로, 데드락 가능성을 줄이기 위해 장바구니 상품을 상품 ID 기준으로 정렬한 뒤 재고를 차감한다.

처리 흐름:

```text
주문 생성 요청
  → 장바구니 상품 조회
  → 상품 ID 기준 정렬
  → ProductService.deductStockWithLock(productId, quantity)
      → ProductRepository.findByIdForUpdate(productId)
      → product.deductStock(quantity)
  → OrderItem 스냅샷 생성
  → Order 저장
  → Payment 사전등록
```

## Consequences

### 장점

* 동일 상품에 대한 동시 주문 시 재고 차감 Race Condition을 방지할 수 있다.
* 재고 부족 검증과 재고 차감을 하나의 트랜잭션 안에서 처리할 수 있다.
* 상품 ID 기준 정렬을 통해 복수 상품 주문 시 데드락 가능성을 완화할 수 있다.
* Product Repository 직접 호출을 OrderFacade에서 제거하고, `ProductService.deductStockWithLock()`으로 위임하여 상품 도메인의 책임을 분리했다.

### 단점 및 주의사항

* 동일 상품에 주문이 집중되면 락 대기 시간이 증가할 수 있다.
* 외부 API 호출이나 긴 작업이 같은 트랜잭션 안에 포함되면 락 보유 시간이 길어질 수 있으므로 주의해야 한다.
* 주문 생성 후 결제 실패 또는 결제대기 주문 취소 시 선차감된 재고를 반드시 복구해야 한다.
* 비관락은 주문 생성 중 재고 차감 경쟁을 막지만, 결제 확정과 주문 취소가 동시에 발생하는 상태 전이 경쟁은 별도 제어가 필요하다.

## 후속 고려사항

* 결제대기 주문 취소 시 선차감 재고 복구 로직 유지
* 결제 실패 또는 보상 트랜잭션 발생 시 재고 복구 처리
* 주문 생성 멱등키 도입 검토

    * 결제 버튼 중복 클릭 또는 네트워크 재시도로 인한 중복 주문 생성 방지
* 주문 취소와 결제 확정이 동시에 발생하는 시나리오에 대한 Order 상태 전이 동시성 제어 검토
* 트래픽 증가 시 Redis 분산 락 또는 재고 예약 테이블 방식 검토
